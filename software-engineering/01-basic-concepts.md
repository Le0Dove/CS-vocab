# 01. 基础概念 (Basic Concepts)

### Abstraction / 抽象
- **中文**：抽象
- **定义**：提取事物本质特征、忽略非关键细节的过程
- **说明**：软件工程核心原则；降低复杂度、提高可维护性；形式：数据抽象、过程抽象、控制抽象；如面向对象中的类和接口
- **相关**：Encapsulation（封装）、Modularity（模块化）、Information Hiding（信息隐藏）

### Agile / 敏捷
- **中文**：敏捷
- **定义**：以迭代、增量方式快速响应需求变化的软件开发方法论
- **说明**：宣言：个体和互动高于流程和工具、可工作的软件高于详尽的文档、客户合作高于合同谈判、响应变化高于遵循计划；代表：Scrum、XP、Kanban
- **相关**：Scrum、Waterfall（瀑布模型）、Iterative Development（迭代开发）

### API (Application Programming Interface) / 应用程序编程接口
- **中文**：应用程序编程接口
- **定义**：软件组件间交互的接口规范，定义可使用的功能、数据格式和协议
- **说明**：类型：REST API、GraphQL、gRPC、Library API；设计原则：简洁、一致、文档完善、向后兼容；是软件集成的基石
- **相关**：Interface（接口）、SDK、Microservices（微服务）

### Architecture / 架构
- **中文**：架构
- **定义**：软件系统的整体结构、组件关系和设计决策的集合
- **说明**：关注点：性能、可扩展性、安全性、可维护性；架构风格：单体、微服务、事件驱动、分层；架构决策影响系统长期演进
- **相关**：Design Pattern（设计模式）、Software Design（软件设计）、System Design（系统设计）

### Artifact / 工件
- **中文**：工件
- **定义**：软件开发过程中产生的任何可交付或可存储的产物
- **说明**：包括：源代码、文档、UML 图、测试用例、编译产物、部署脚本；配置管理负责版本控制
- **相关**：Deliverable（可交付物）、Version Control（版本控制）、Build（构建）

### CASE Tool / CASE 工具
- **中文**：计算机辅助软件工程工具
- **全称**：Computer-Aided Software Engineering
- **定义**：支持软件开发、维护、管理的自动化工具
- **说明**：类型：设计工具（UML 工具）、代码生成器、测试工具、项目管理工具；代表：Visual Paradigm、Enterprise Architect、Rational Rose
- **相关**：UML、IDE、Software Development（软件开发）

### Component / 组件
- **中文**：组件
- **定义**：系统中可独立部署、替换的模块化软件单元
- **说明**：通过接口与其他组件交互；实现高内聚、低耦合；组件化开发提高复用性；如 COM、EJB、Web Components
- **相关**：Module（模块）、Interface（接口）、Component-Based Development（基于组件的开发）

### Configuration Management / 配置管理
- **中文**：配置管理
- **定义**：管理软件变更、版本、构建和发布的过程
- **说明**：包括：版本控制、变更控制、构建管理、发布管理；工具：Git、SVN、Perforce；确保软件完整性和可追溯性
- **相关**：Version Control（版本控制）、Change Management（变更管理）、CI/CD

### Coupling / 耦合
- **中文**：耦合
- **定义**：模块之间相互依赖的程度
- **说明**：低耦合是设计目标；类型：数据耦合（低）、标记耦合、控制耦合、外部耦合、公共耦合、内容耦合（高）；降低耦合提高模块独立性
- **相关**：Cohesion（内聚）、Modularity（模块化）、Dependency（依赖）

### Cohesion / 内聚
- **中文**：内聚
- **定义**：模块内部元素之间相互关联的程度
- **说明**：高内聚是设计目标；类型：功能内聚（高）、顺序内聚、通信内聚、过程内聚、时间内聚、逻辑内聚、偶然内聚（低）；高内聚使模块职责单一
- **相关**：Coupling（耦合）、Single Responsibility Principle（单一职责原则）、Modularity（模块化）

### Dependency / 依赖
- **中文**：依赖
- **定义**：一个模块需要另一个模块的功能或数据才能正确运行的关系
- **说明**：管理依赖是软件工程关键；依赖注入（DI）降低耦合；依赖管理工具：Maven、npm、pip、Gradle；循环依赖应避免
- **相关**：Coupling（耦合）、Dependency Injection（依赖注入）、Module（模块）

### Deployment / 部署
- **中文**：部署
- **定义**：将软件系统安装、配置到目标环境并使其可用的过程
- **说明**：环境：开发、测试、预发布（Staging）、生产（Production）；策略：蓝绿部署、金丝雀发布、滚动更新；自动化部署减少人为错误
- **相关**：Release（发布）、CI/CD、DevOps、Environment（环境）

### Documentation / 文档
- **中文**：文档
- **定义**：描述软件系统需求、设计、实现、使用和运维的书面材料
- **说明**：类型：需求文档、设计文档、API 文档、用户手册、运维文档；工具：Markdown、Swagger、Javadoc、Confluence；好文档降低沟通成本
- **相关**：Specification（规格说明）、README、Wiki

### Encapsulation / 封装
- **中文**：封装
- **定义**：将数据和操作数据的方法绑定在一起、并隐藏内部实现细节的原则
- **说明**：通过访问修饰符（public/private/protected）控制可见性；保护数据完整性；是面向对象三大特性之一（封装、继承、多态）
- **相关**：Abstraction（抽象）、Information Hiding（信息隐藏）、OOP

### Framework / 框架
- **中文**：框架
- **定义**：提供通用功能和结构的软件平台，开发者在其上构建特定应用
- **说明**：与库（Library）不同：框架控制流程（IoC），库被代码调用；代表：Spring、Django、React、Vue、Ruby on Rails；提高开发效率
- **相关**：Library（库）、Design Pattern（设计模式）、Inversion of Control（控制反转）

### Information Hiding / 信息隐藏
- **中文**：信息隐藏
- **定义**：将模块的实现细节对其他模块隐藏、只暴露必要接口的设计原则
- **说明**：David Parnas 提出；降低系统复杂度；支持模块化开发和独立维护；是封装和抽象的基础
- **相关**：Encapsulation（封装）、Abstraction（抽象）、Modularity（模块化）

### Interface / 接口
- **中文**：接口
- **定义**：定义组件或模块间交互契约的规范，不包含实现
- **说明**：形式：API、GUI、硬件接口、编程语言接口（Java interface）；接口隔离原则（ISP）：客户端不应依赖不用的接口
- **相关**：API、Abstraction（抽象）、Contract（契约）

### Iterative Development / 迭代开发
- **中文**：迭代开发
- **定义**：将项目分解为多个短周期（迭代），每个迭代交付可用增量的开发方式
- **说明**：每个迭代包含需求、设计、编码、测试；快速反馈、降低风险；与增量开发、敏捷开发密切相关
- **相关**：Agile（敏捷）、Increment（增量）、Sprint（冲刺）

### Library / 库
- **中文**：库
- **定义**：可重用的代码集合，供程序调用以实现特定功能
- **说明**：与框架不同：库被代码调用，框架调用代码；静态库（编译时链接）、动态库（运行时加载）；代表：标准库、第三方库
- **相关**：Framework（框架）、Module（模块）、API

### Lifecycle / 生命周期
- **中文**：生命周期
- **定义**：软件从概念到退役的完整过程
- **说明**：阶段：需求分析 → 设计 → 实现 → 测试 → 部署 → 运维 → 退役；模型：瀑布、V 模型、螺旋、敏捷；不同模型适用于不同项目
- **相关**：SDLC、Waterfall（瀑布模型）、Software Process（软件过程）

### Maintainability / 可维护性
- **中文**：可维护性
- **定义**：软件被修改、修复、扩展的难易程度
- **说明**：子特性：可分析性、可修改性、稳定性、可测试性；受代码质量、文档、架构影响；可维护性成本占软件总成本的 60% 以上
- **相关**：Technical Debt（技术债务）、Code Quality（代码质量）、Refactoring（重构）

### Methodology / 方法论
- **中文**：方法论
- **定义**：指导软件开发的一套系统化的方法、原则和实践
- **说明**：传统：瀑布模型、V 模型、螺旋模型；现代：敏捷（Scrum、XP）、精益、DevOps；选择取决于项目特点和团队文化
- **相关**：Agile（敏捷）、Waterfall（瀑布模型）、Software Process（软件过程）

### Modularity / 模块化
- **中文**：模块化
- **定义**：将系统分解为独立、可替换的模块的设计原则
- **说明**：高内聚、低耦合；模块有清晰接口；好处：易于理解、开发、测试、维护、复用；是应对复杂性的基本策略
- **相关**：Component（组件）、Coupling（耦合）、Cohesion（内聚）

### Module / 模块
- **中文**：模块
- **定义**：系统中完成特定功能、相对独立的代码单元
- **说明**：模块有明确边界和接口；粒度：函数、类、包、服务；模块化是软件设计的核心；模块间通过接口通信
- **相关**：Component（组件）、Modularity（模块化）、Interface（接口）

### OOP (Object-Oriented Programming) / 面向对象编程
- **中文**：面向对象编程
- **定义**：以对象（数据和行为的封装体）为基本单位组织程序的编程范式
- **说明**：四大特性：封装、继承、多态、抽象；设计原则：SOLID；代表语言：Java、C++、Python、C#
- **相关**：Class（类）、Object（对象）、Design Pattern（设计模式）

### Paradigm / 范式
- **中文**：范式
- **定义**：编程或软件设计的基本风格和思维方式
- **说明**：编程范式：面向对象（OOP）、函数式（FP）、过程式、逻辑式、并发式；架构范式：分层、事件驱动、微服务；不同范式解决不同问题
- **相关**：OOP、Functional Programming（函数式编程）、Software Design（软件设计）

### Portability / 可移植性
- **中文**：可移植性
- **定义**：软件从一种环境迁移到另一种环境的难易程度
- **说明**：影响因素：编程语言、依赖库、操作系统、硬件架构；提高方法：使用标准库、抽象平台差异、容器化（Docker）；与兼容性相关
- **相关**：Compatibility（兼容性）、Platform（平台）、Docker

### Quality Assurance (QA) / 质量保证
- **中文**：质量保证
- **定义**：确保软件满足质量要求的系统性活动
- **说明**：包括：流程定义、标准制定、审计、评审、测试；与 QC（质量控制）不同：QA 关注流程，QC 关注产品；预防胜于检测
- **相关**：QC（质量控制）、Testing（测试）、Software Quality（软件质量）

### Refactoring / 重构
- **中文**：重构
- **定义**：在不改变外部行为的前提下改进代码内部结构的技术
- **说明**：目的：提高可读性、降低复杂度、便于扩展；常见重构：提取函数、重命名、消除重复、简化条件；工具支持：IDE 重构功能
- **相关**：Technical Debt（技术债务）、Code Smell（代码异味）、Clean Code（整洁代码）

### Requirement / 需求
- **中文**：需求
- **定义**：软件系统必须满足的功能、性能、约束条件
- **说明**：分类：功能需求（做什么）、非功能需求（性能/安全/可用性）；来源：用户、业务、法规；管理：需求获取、分析、规格说明、变更控制
- **相关**：SRS（软件需求规格说明）、Use Case（用例）、Requirement Engineering（需求工程）

### Reusability / 可复用性
- **中文**：可复用性
- **定义**：软件组件被用于多个系统或上下文的程度
- **说明**：复用级别：代码（函数/类）、设计（模式）、架构（框架）、需求；好处：减少开发时间、提高质量；挑战：通用性与专用性的平衡
- **相关**：Library（库）、Framework（框架）、Component（组件）

### Scalability / 可扩展性
- **中文**：可扩展性
- **定义**：软件系统在负载增加时保持性能的能力
- **说明**：垂直扩展（Scale Up）：增强单机性能；水平扩展（Scale Out）：增加机器数量；架构设计需考虑数据分区、负载均衡、缓存
- **相关**：Performance（性能）、Load Balancing（负载均衡）、Architecture（架构）

### SDLC (Software Development Life Cycle) / 软件开发生命周期
- **中文**：软件开发生命周期
- **定义**：软件从规划到退役的完整过程框架
- **说明**：阶段：计划 → 需求分析 → 设计 → 编码 → 测试 → 部署 → 维护；模型：瀑布、迭代、敏捷、DevOps；选择取决于项目需求
- **相关**：Lifecycle（生命周期）、Waterfall（瀑布模型）、Agile（敏捷）

### Service / 服务
- **中文**：服务
- **定义**：对外提供特定功能的独立软件单元
- **说明**：SOA（面向服务的架构）：服务通过标准协议通信；微服务：细粒度、独立部署的服务；服务间通过 API 或消息队列通信
- **相关**：Microservices（微服务）、SOA、API

### Software Crisis / 软件危机
- **中文**：软件危机
- **定义**：20 世纪 60-70 年代软件项目普遍超预算、延期、质量低下的现象
- **说明**：原因：需求不明确、规模扩大、缺乏系统方法；催生了软件工程学科（1968 年 NATO 会议正式提出）；推动了结构化方法和软件工程教育
- **相关**：Software Engineering（软件工程）、Software Process（软件过程）

### Software Engineering / 软件工程
- **中文**：软件工程
- **定义**：系统化、规范化、可量化的软件开发与维护方法
- **说明**：知识领域：需求、设计、构建、测试、维护、配置管理、工程管理、过程、质量；IEEE 定义：将系统化、规范化、可量化的方法应用于软件开发
- **相关**：Computer Science（计算机科学）、SDLC、Agile（敏捷）

### Specification / 规格说明
- **中文**：规格说明
- **定义**：对软件系统需求、设计或接口的精确、详细的描述文档
- **说明**：需求规格（SRS）、设计规格、接口规格；形式：自然语言、结构化语言、形式化规格（Z 语言、B 方法）；是开发和测试的依据
- **相关**：Requirement（需求）、SRS、Documentation（文档）

### SRS (Software Requirements Specification) / 软件需求规格说明
- **中文**：软件需求规格说明
- **定义**：详细描述软件功能需求和非功能需求的正式文档
- **说明**：内容包括：功能需求、性能需求、接口需求、约束、验收标准；模板：IEEE 830；是开发、测试、验收的基准
- **相关**：Requirement（需求）、Specification（规格说明）、Use Case（用例）

### Stakeholder / 利益相关者
- **中文**：利益相关者
- **定义**：对软件项目有影响或受项目影响的个人或组织
- **说明**：包括：用户、客户、开发团队、测试团队、运维、管理层、最终用户；识别和管理利益相关者是项目成功的关键
- **相关**：Requirement（需求）、Project Management（项目管理）、User（用户）

### Subsystem / 子系统
- **中文**：子系统
- **定义**：大型系统中相对独立、完成特定功能的分系统
- **说明**：子系统有自己的模块和接口；通过接口与其他子系统交互；降低系统整体复杂度；如电商系统的订单子系统、支付子系统
- **相关**：System（系统）、Module（模块）、Component（组件）

### System / 系统
- **中文**：系统
- **定义**：由相互关联的组件组成、共同完成特定功能的整体
- **说明**：软件系统包括：应用程序、数据库、中间件、基础设施；系统工程关注整体而非局部；系统思维是架构师的核心能力
- **相关**：Architecture（架构）、Subsystem（子系统）、Component（组件）

### Use Case / 用例
- **中文**：用例
- **定义**：描述系统如何与外部参与者交互以实现特定目标的场景
- **说明**：用例图（Use Case Diagram）展示参与者和用例关系；包含：参与者、前置条件、基本流、扩展流、后置条件；是需求分析的重要工具
- **相关**：UML、Requirement（需求）、Actor（参与者）

### Waterfall Model / 瀑布模型
- **中文**：瀑布模型
- **定义**：线性顺序的软件开发模型，每个阶段完成后才进入下一阶段
- **说明**：阶段：需求 → 设计 → 实现 → 验证 → 维护；优点：简单、阶段明确；缺点：难以适应需求变化、反馈延迟；适合需求明确的项目
- **相关**：SDLC、Agile（敏捷）、V-Model（V 模型）

### White Box / 白盒
- **中文**：白盒
- **定义**：基于内部结构和实现细节进行测试或分析的方法
- **说明**：白盒测试：覆盖代码路径、分支、条件；需要了解内部实现；与黑盒测试相对；工具：代码覆盖率工具、静态分析工具
- **相关**：Black Box（黑盒）、Testing（测试）、Code Coverage（代码覆盖率）
